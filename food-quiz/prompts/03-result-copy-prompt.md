# 味觉灵魂测试 — 结果文案生成提示词

> **使用方法**：把整篇喂给任何 LLM（Claude / GPT / Gemini / 国产模型均可）。
> 想生成某一类时只保留对应章节即可。LLM 应严格按本规约输出 JSON。

---

## 项目背景（必读）

「味觉灵魂测试」是一个纯前端中国味觉性格测试。用户答 30–45 道题 → 得到 8 维味觉向量 → 看到一份图文并茂的结果页。结果页消费 **5 类文案**（已下线 1 类避雷指南），全部以静态 JSON 形式落盘，运行时按规则匹配。

### 8 维味觉体系（**唯一真相源，全篇通用**）

| 位 | 维度 | 字段名 | 单字母 | 中文 |
|:--:|:--|:--|:--:|:--|
| 0 | 酸 | `sour` | **S** | 酸 |
| 1 | 甜 | `sweet` | **T** | 甜 |
| 2 | 苦 | `bitter` | **K** | 苦 |
| 3 | 辣 | `spicy` | **L** | 辣 |
| 4 | 咸 | `salty` | **I** | 咸（取第 2 字母 i，因「浓」占 X） |
| 5 | 浓 | `rich` | **X** | 浓（**绝不写成"鲜"或 umami**） |
| 6 | 脆 | `crunchy` | **C** | 脆 |
| 7 | 嫩 | `tender` | **N** | 嫩 |

### 大小写编码

- **大写 = 高档（归一化值 > 60）**，**小写 = 低档（≤ 60）**。极档（≥ 90）不改变大小写，只额外触发极档文案。
- 索引串顺序固定 `S T K L I X C N`，例如 `StKliXcN` 表示「高酸 / 高苦 / 高浓」3 维高档，其余低档。
- 每串映射一个 0–255 的整数：S 是 bit7（MSB）… N 是 bit0；大写=1，小写=0。例如 `StKliXcN`=10100100=164。

### 档位文案标签规则

每个维度按 profile 值分三档（仅文案层面用，与上面"大小写编码"是两套并行系统）：

- 维度名前缀：`低{维}` / `重{维}` / `重{维} ⚡极`
- **浓维（X）特例**：`清淡` / `浓` / `口味重 ⚡极`

例：spicy=70 → `重辣`；rich=95 → `口味重 ⚡极`；sour=20 → `低酸`。

## 风格规约（**所有文案通用，违反即返工**）

### 语言

1. **中文为主**，偶尔点缀 emoji（🔥 🍜 🌶️ 🥢 等，每条不超 1 个，能不用就不用）
2. **略带书面感的人物画像风**——像在描写一个朋友的性格，不是吐槽。基础句式参考：
   - 「你是……的人」
   - 「你大概……」
   - 「在你看来，……」
   - 「这种偏好让你……」
3. 可以**轻微调侃**或带一点反讽，但**不要油腻、不要过度网络梗**
4. **每条 18–45 字**为佳，区间文案可到 55 字
5. 标签 `label` **3–8 字**

### 风格示例对比（重要！）

| ❌ 太吐槽（避免） | ✅ 略书面画像感（采纳） |
|:---|:---|
| 你的胃已经辞职了但你还在加辣 | 你是那种没有辣椒就吃不下饭的人，越辣越觉得清醒 |
| 别人怕辣，你怕不辣 | 在你看来，辣不是味道，是一种节奏 |
| 酸到牙倒还在嗦 | 你是嗜酸的人，酸味让你觉得食物是活的 |
| 朋友选不出餐厅找你准没错 | 你是那种什么都吃得惯的人，餐桌上很少给别人添麻烦 |

### 反 AI 味（严格执行）

- **不要破折号** `—— / – / —`（中文里这是 AI 写作的标志性痕迹，用句号或逗号代替）
- **不要"在……中" "在……上" "于……之间"**这类 AI 套话
- **不要"是 X 的代表" "堪称 Y 的典范" "称得上是……"**这类空洞总结
- **不要排比三连**：「你是 X，你是 Y，你是 Z」这种套路
- **不要"-ing 式肤浅分析**：「不断追求着……」「持续探索着……」「始终保持着……」
- **不要「让……成为」「成就了……」「定义了……」**这种伪文学翻译腔
- **不要括号注解**：「（你懂的）」「（懂得都懂）」
- **不要"无论……都" "不管……总是"**这种放之四海皆准的废话句式
- 中文标点要**全角**（。，！？；：），不要混用半角

### 内容

- 描述**用户的口味性格**，不是描述食物本身
- 用**第二人称"你"**作为主语
- 可以**带一点观察感、共情感**，像作者在描摹一个具体的人
- 避免**性别 / 地域 / 体型 / 年龄**的刻板印象
- 不要医疗 / 营养建议
- 不要广告 / 品牌植入

### 输出格式

- 严格 JSON
- UTF-8、缩进 2 空格、末尾换行
- 不要在 JSON 里写注释

## 五类文案逐条规约

### 1. 区间文案 `intervals/{000..255}.json`（256 条）

**触发**：永远显示。运行时根据 8 维高低位组合算出 0–255 的索引，匹配对应文件。

**单文件 schema**：
```json
{
  "index": 164,
  "key": "StKliXcn",
  "label": "酸苦浓三重奏",
  "copy": "你是那种偏爱浓烈口味的人，酸、苦、浓三种味道都能接受，清淡的菜在你这里很难留下印象。"
}
```

**字段语义**：
- `index`：0–255，与文件名（如 `164.json`）必须一致
- `key`：8 字符大小写串，与 index 二进制位对应（S=bit7…N=bit0）
- `label`：3–8 字标签，**反映高档维度的组合特征**
- `copy`：一句话画像描写，18–55 字

**生成原则**：
- 高档维度数量决定语调：
  - **0 个高档**（`stklixcn`）：「你是那种对食物没什么执念的人，什么都能吃，什么都谈不上偏爱」
  - **1 个高档**：聚焦那一个维度，写出它在用户性格里的位置
  - **2–3 个高档**：写出维度组合的化学反应，落到具体的口味场景
  - **4–6 个高档**：「你是那种重口味爱好者，一桌菜里总会挑最浓烈的几道」
  - **7–8 个高档**：「你是真正意义上什么都吃的人，从极酸到极辣都能欣赏」
- **256 条不能用同一个模板**——这是当前最大问题。每条至少要在 label 或 copy 文案上有独特性
- 高档的**组合**可以构成具体联想：
  - `STkl..` 酸甜 → 糖醋家族 / 奶茶人格
  - `klIX..` 辣咸浓 → 川渝灵魂 / 下饭刺客
  - `KL....` 苦辣 → 黑咖啡配麻辣火锅，你大概是个心口都重的人
- 浓维 X 高档时，文案侧重"分量重 / 厚味"，不要写成"鲜"

**重要**：256 条全要生成。可以分批，每批一定数量并保持 label 多样。

### 2. 极档文案 `extreme/{s,t,k,l,i,x,c,n}.json`（8 条）

**触发**：某维归一化值 ≥ 90，叠加在区间文案之外。

**单文件 schema**：
```json
{
  "dim": "spicy",
  "letter": "l",
  "label": "极辣",
  "threshold": 90,
  "copy": [
    "你是那种没有辣椒就吃不下饭的人，越辣越觉得清醒",
    "在你看来，辣不是味道，是一种节奏，是让食物活过来的方式",
    "辣对你来说几乎是一种信仰，清淡的菜在你这里很难称得上一顿饭"
  ]
}
```

**字段**：
- `dim`：英文字段名（sour/sweet/bitter/spicy/salty/rich/crunchy/tender）
- `letter`：单字母小写（s/t/k/l/i/x/c/n）
- `label`：`极{维}`，浓维写 `极浓`
- `threshold`：固定 90
- `copy`：**3–5 条**，每条 20–40 字，运行时随机抽 1 条

**生成原则**：
- 极档=用户在该维度走到了极致，文案可以**带一点反讽或观察感**，但保持画像式
- 8 条之间气质要差异化：
  - 极酸（s）：你是嗜酸的人，酸味让食物在你这里有了棱角
  - 极甜（t）：你的口味里始终有一份对甜的执念
  - 极苦（k）：你是那种把苦味当作正餐的人，清淡的甜在你这里反而无聊
  - 极辣（l）：见上方样本
  - 极咸（i）：你是那种酱油可以倒一勺的人，咸味是你下饭的底线
  - 极浓（x）：你偏爱厚重的口味，清汤寡水的菜在你这里很难成立
  - 极脆（c）：脆是你判断一道菜值不值的标准之一
  - 极嫩（n）：你大概是嫩派，咬不动的食物对你而言已经输了一半

### 3. 联动文案 `synergies/{a-b}.json`（10 对 + 1 个兜底）

**触发**：Top1 与 Top2 维度都 > 60。运行时按字母对查找；未命中走 `_fallback.json` 模板。

**已存在的字母对**：c-k, c-n, i-l, k-t, l-s, l-x, n-x, s-t, s-x, t-x（10 对）。

**单文件 schema**：
```json
{
  "pair": ["l", "x"],
  "letters": ["l", "x"],
  "label": "辣浓双煞",
  "copy": [
    "你是辣与浓都要的人，川渝火锅那一锅红油大概最懂你",
    "辣味遇上厚味，对你来说是一种饱满的双重满足",
    "你大概很难被一碗清汤寡水的面打动，要的就是热烈又厚重的那一口"
  ]
}
```

**字段**：
- `pair` / `letters`：两个字母，**字母字典序排列**（`l-x` 不是 `x-l`）
- `label`：4–6 字组合昵称，例 `酸甜双煞`、`脆嫩二重奏`、`苦辣同行`
- `copy`：**2–4 条**，每条 20–40 字

**`_fallback.json`**（命中不到具体对时用）：
```json
{
  "id": "_fallback",
  "label": "强强联合",
  "copy": [
    "你的两个最强维度同时在线，组合出了一种很有辨识度的口味",
    "这两个维度同时拉满，让你对食物的偏好比一般人更清晰"
  ]
}
```

**生成原则**：
- 字母对要写出**具体食物 / 菜系联想**，落到画像里：
  - l-x（辣浓）→ 川渝火锅、毛血旺、那一锅红油
  - s-t（酸甜）→ 糖醋、菠萝咕咾、菠萝古老肉
  - k-t（苦甜）→ 黑巧克力、抹茶拿铁、苦尽甘来的咖啡
  - i-l（咸辣）→ 东北烧烤、麻辣烫、烤串那一撮辣椒面
  - c-n（脆嫩）→ 外脆内嫩、生煎包、酥皮里裹着流心的那种
  - s-x（酸浓）→ 老坛酸菜、东北酸菜锅、那一口酸得有底气
  - t-x（甜浓）→ 红烧肉那一勺浓汁、糖醋排骨、上海本帮菜的甜
  - n-x（嫩浓）→ 滑蛋、嫩牛肉炖、入口即化的那一份厚味
  - l-s（辣酸）→ 酸辣粉、贵州酸汤鱼、又酸又辣的开胃小菜
  - c-k（脆苦）→ 苦瓜炒蛋、咖啡豆、带点苦的脆才有层次

### 4. 全能文案 `allround/{01..04}.json`（4 条 + `_index.json`）

**触发**：8 维标准差 std < 15（用户各维度都比较均衡）。**替换** 256 区间文案分支。

**单文件 schema**：
```json
{
  "id": "01",
  "label": "随和派食客",
  "copy": [
    "你是那种什么都吃得惯的人，餐桌上很少给别人添麻烦，朋友约饭最先想到你",
    "8 维各自温和，没有特别突出的偏好，也没有特别讨厌的味道",
    "你大概是那种愿意尝试任何菜的人，对食物保持着不挑剔的好奇"
  ]
}
```

**`_index.json`**（运行时随机抽 1 个文件）：
```json
{
  "module": "allround",
  "id": "_index",
  "ids": ["01", "02", "03", "04"]
}
```

**字段**：
- `id`：`01` / `02` / `03` / `04`
- `label`：4–8 字昵称，例 `随和派食客`、`见多识广的胃`、`心情型味觉`、`好奇心驱动者`
- `copy`：**3 条**，每条 20–45 字，运行时随机抽 1 条

**生成原则**：
- 4 个文件代表**4 种"全能"性格切面**：
  - 01：随和派——什么都吃得惯，不挑食的那种
  - 02：见多识广——什么都吃过，对食物有阅历感
  - 03：心情型——今天想吃什么就吃什么，跟着情绪走
  - 04：好奇心驱动——愿意尝新，对没吃过的菜有兴趣
- 4 个 label 要明显不同，不要全部都"什么都好吃"

## 输出方式

### 选项 A：一次性输出全部 5 类（推荐）

按下面顺序输出，**每个 JSON 文件之间用 `---FILE: 路径.json---` 分隔符**：

```
---FILE: intervals/000.json---
{...}
---FILE: intervals/001.json---
{...}
...
---FILE: extreme/s.json---
{...}
...
---FILE: synergies/_fallback.json---
{...}
---FILE: synergies/l-x.json---
{...}
...
---FILE: allround/_index.json---
{...}
---FILE: allround/01.json---
{...}
...
```

用户用脚本切分（grep `---FILE:` 拆文件）。

### 选项 B：分批生成

如果一次输出太长（256 条 intervals 是大头），可以这样组织：

1. 先输出全部 **8 条 extreme + 11 条 synergies + 5 条 allround = 24 个文件**
2. 再分 4 批输出 intervals：`000-063`、`064-127`、`128-191`、`192-255`，每批 64 条

每批末尾要附一行 `---DONE: <类别>---` 表示该类完成。
