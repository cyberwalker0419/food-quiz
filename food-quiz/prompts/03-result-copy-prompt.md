# 味觉灵魂测试 — 结果文案生成提示词

> **使用方法**：把整篇喂给任何 LLM（Claude / GPT / Gemini / 国产模型均可）。
> 想生成某一类时只保留对应章节即可。LLM 应严格按本规约输出 JSON。

---

## 项目背景（必读）

「味觉灵魂测试」是一个纯前端中国味觉性格测试。用户答 25–45 道题 → 得到 8 维味觉向量 → 看到一份图文并茂的结果页。结果页消费 **5 类文案**（已下线 1 类避雷指南），全部以静态 JSON 形式落盘，运行时按规则匹配。

### 8 维味觉体系（**唯一真相源，全篇通用**）

| 位 | 维度 | 字段名 | 单字母 | 中文 |
|:--:|:--|:--|:--:|:--|
| 0 | 酸 | `sour` | **S** | 酸 |
| 1 | 甜 | `sweet` | **T** | 甜 |
| 2 | 苦 | `bitter` | **K** | 苦 |
| 3 | 辣 | `spicy` | **L** | 辣 |
| 4 | 咸 | `salty` | **I** | 咸（取第 2 字母 i，因「浓」占 X） |
| 5 | 浓 | `rich` | **X** | 浓（**绝不写成"鲜"或 umami**） |
| 6 | 脆 | `crunchy` | **C** | 脆 |
| 7 | 嫩 | `tender` | **N** | 嫩 |

### 大小写编码

- **大写 = 高档（归一化值 > 60）**，**小写 = 低档（≤ 60）**。极档（≥ 90）不改变大小写，只额外触发极档文案。
- 索引串顺序固定 `S T K L I X C N`，例如 `StKliXcN` 表示「高酸 / 高苦 / 高浓」3 维高档，其余低档。
- 每串映射一个 0–255 的整数：S 是 bit7（MSB）… N 是 bit0；大写=1，小写=0。例如 `StKliXcN`=10100100=164。

### 档位文案标签规则

每个维度按 profile 值分三档（仅文案层面用，与上面"大小写编码"是两套并行系统）：

- 维度名前缀：`低{维}` / `重{维}` / `重{维} ⚡极`
- **浓维（X）特例**：`清淡` / `浓` / `口味重 ⚡极`

例：spicy=70 → `重辣`；rich=95 → `口味重 ⚡极`；sour=20 → `低酸`。

## 风格规约（**所有文案通用，违反即返工**）

### 语言

1. **中文为主**，可点缀少量 emoji（🔥 🍜 🌶️ 🥢 等，每条不超 2 个）
2. **口语化、有梗**，像朋友吐槽，不要严肃说教
3. **每条 15–40 字**为佳，区间文案可到 50 字
4. 标签 `label` **3–8 字**

### 反 AI 味（重要）

- **不要破折号** `—— / – / —`（中文里这是 AI 写作的标志性痕迹）
- **不要"在……中" "在……上"**这类 AI 套话
- **不要"是 X 的代表" "堪称 Y 的典范"**这类空洞结论
- **不要排比三连**："你是 X，你是 Y，你是 Z" 这种套路
- **不要"-ing 式肤浅分析**："不断追求着……、持续探索着……"
- **不要「让……成为」、「成就了……」**这种伪文学翻译腔
- **不要括号括起来的注解**：「（你懂的）」「（懂得都懂）」
- 中文标点要**全角**（。，！？；：），不要混用半角

### 内容

- 描述用户**口味性格**，不是描述食物
- 可以**轻微调侃**用户（"你的胃已经辞职了但你还在加辣"），不能羞辱
- 避免**性别 / 地域 / 体型 / 年龄**的刻板印象
- 不要医疗 / 营养建议
- 不要广告 / 品牌植入

### 输出格式

- 严格 JSON
- UTF-8、缩进 2 空格、末尾换行
- 不要在 JSON 里写注释

## 五类文案逐条规约

### 1. 区间文案 `intervals/{000..255}.json`（256 条）

**触发**：永远显示。运行时根据 8 维高低位组合算出 0–255 的索引，匹配对应文件。

**单文件 schema**：
```json
{
  "index": 164,
  "key": "StKliXcn",
  "label": "酸苦浓三重奏",
  "copy": "酸、苦、浓三个重口味同时拉满，你不爱清淡，要的就是这一口浓烈。"
}
```

**字段语义**：
- `index`：0–255，与文件名（如 `164.json`）必须一致
- `key`：8 字符大小写串，与 index 二进制位对应（S=bit7…N=bit0）
- `label`：3–8 字标签，**反映高档维度的组合特征**
- `copy`：一句话描述，15–50 字

**生成原则**：
- 高档维度数量决定语调：
  - **0 个高档**（`stklixcn`）：「味觉素人 / 什么都行的随和派」
  - **1 个高档**：聚焦那一个维度的性格特征
  - **2–3 个高档**：写出维度组合的化学反应
  - **4–6 个高档**：「重口味探路者」类气质
  - **7–8 个高档**：「味觉深渊老饕」「无所不吃」
- **256 条不能用同一个模板**——这是当前最大问题。每条至少要在 label 或 copy 文案上有独特性。
- 高档的**组合**可以构成新词：
  - `STkl..` 酸甜 → 「糖醋家族」「奶茶人格」
  - `klIX..` 辣咸浓 → 「重庆灵魂」「下饭刺客」
  - `KL....` 苦辣 → 「黑咖啡 + 麻辣火锅，你心里苦」
- 浓维 X 高档时，文案可侧重"分量重 / 厚味"，不要写成"鲜"

**重要**：256 条全要生成。可以分批，每批一定数量并保持 label 多样。

### 2. 极档文案 `extreme/{s,t,k,l,i,x,c,n}.json`（8 条）

**触发**：某维归一化值 ≥ 90，叠加在区间文案之外。

**单文件 schema**：
```json
{
  "dim": "spicy",
  "letter": "l",
  "label": "极辣",
  "threshold": 90,
  "copy": [
    "🔥 辣到冒汗也要继续，辣椒是你的信仰",
    "你的胃已经提交了辞呈，但你还在加辣",
    "别人怕辣，你怕不辣"
  ]
}
```

**字段**：
- `dim`：英文字段名（sour/sweet/bitter/spicy/salty/rich/crunchy/tender）
- `letter`：单字母小写（s/t/k/l/i/x/c/n）
- `label`：`极{维}`，浓维写 `极浓`
- `threshold`：固定 90
- `copy`：**3–5 条**，每条 15–35 字，运行时随机抽 1 条

**生成原则**：
- 极档=用户在该维度走到了极致，文案可以**夸张吐槽**
- 8 条之间气质要差异化：
  - 极酸（s）：酸到牙倒、东北榨菜永远多备一罐
  - 极甜（t）：糖罐人格、奶茶续命
  - 极苦（k）：黑咖啡续命、苦瓜原教旨
  - 极辣（l）：变态辣不眨眼、马爹儿没有最辣只有更辣
  - 极咸（i）：钠摄入超标届的卷王、酱油倒一勺
  - 极浓（x）：要的就是那口厚重、清汤寡水会失眠
  - 极脆（c）：薯片之友、咬不动的不吃
  - 极嫩（n）：入口即化才是正义、嚼劲？不存在的

### 3. 联动文案 `synergies/{a-b}.json`（10 对 + 1 个兜底）

**触发**：Top1 与 Top2 维度都 > 60。运行时按字母对查找；未命中走 `_fallback.json` 模板。

**已存在的字母对**：c-k, c-n, i-l, k-t, l-s, l-x, n-x, s-t, s-x, t-x（10 对）。

**单文件 schema**：
```json
{
  "pair": ["l", "x"],
  "letters": ["l", "x"],
  "label": "鲜辣双煞",
  "copy": [
    "🔥 鲜辣双煞，味觉的火上浇油组合",
    "辣椒遇见浓味，舌头在开派对",
    "你的口味是火锅底料级别的"
  ]
}
```

**字段**：
- `pair` / `letters`：两个字母，**字母字典序排列**（`l-x` 不是 `x-l`）
- `label`：4–6 字组合昵称，例 `酸甜双煞`、`脆嫩二重奏`、`苦辣同行`
- `copy`：**2–4 条**，每条 15–35 字

**`_fallback.json`**（命中不到具体对时用）：
```json
{
  "id": "_fallback",
  "label": "强强联合",
  "copy": [
    "你的两个最强维度正在组队，味觉界的新组合",
    "双维度共振，让你的口味体验再升级"
  ]
}
```

**生成原则**：
- 字母对要写出**具体食物 / 菜系联想**：
  - l-x（辣浓）→ 川渝火锅、毛血旺
  - s-t（酸甜）→ 糖醋、菠萝咕咾
  - k-t（苦甜）→ 黑巧克力、抹茶拿铁
  - i-l（咸辣）→ 东北小烧烤、麻辣烫
  - c-n（脆嫩）→ 软糯外脆、生煎包
  - s-x（酸浓）→ 老坛酸菜、东北酸菜锅
  - t-x（甜浓）→ 红烧肉、糖醋排骨那一勺浓汁
  - n-x（嫩浓）→ 滑蛋、嫩牛肉炖
  - l-s（辣酸）→ 酸辣粉、贵州酸汤
  - c-k（脆苦）→ 苦瓜炒蛋、咖啡豆
- 别再补「什么甜苦」「酸咸」这种不接地气的对（除非有具体菜联想）

### 4. 全能文案 `allround/{01..04}.json`（4 条 + `_index.json`）

**触发**：8 维标准差 std < 15（用户各维度都比较均衡）。**替换** 256 区间文案分支。

**单文件 schema**：
```json
{
  "id": "01",
  "label": "味觉端水大师",
  "copy": [
    "什么都能吃，什么都好吃，你就是餐桌上的和事佬",
    "8 维均匀分布，堪称味觉界的端水艺术家",
    "朋友聚餐选不出餐厅？找你准没错"
  ]
}
```

**`_index.json`**（运行时随机抽 1 个文件）：
```json
{
  "module": "allround",
  "files": ["01.json", "02.json", "03.json", "04.json"]
}
```

**字段**：
- `id`：`01` / `02` / `03` / `04`
- `label`：4–8 字昵称，例 `味觉端水大师`、`随和派食客`、`百搭型胃`
- `copy`：**3 条**，运行时随机抽 1 条

**生成原则**：
- 4 个文件代表**4 种"全能"性格切面**：
  - 01：和事佬 / 不挑食型
  - 02：见多识广 / 老饕级
  - 03：随性 / 心情党
  - 04：好奇心强 / 探索型
- 不要全部都"什么都好吃"

## 输出方式

### 选项 A：一次性输出全部 5 类（推荐）

按下面顺序输出，**每个 JSON 文件之间用 `---FILE: 路径.json---` 分隔符**：

```
---FILE: intervals/000.json---
{...}
---FILE: intervals/001.json---
{...}
...
---FILE: extreme/s.json---
{...}
...
---FILE: synergies/_fallback.json---
{...}
---FILE: synergies/l-x.json---
{...}
...
---FILE: allround/_index.json---
{...}
---FILE: allround/01.json---
{...}
...
```

用户用脚本切分（grep `---FILE:` 拆文件）。

### 选项 B：分批生成

如果一次输出太长（256 条 intervals 是大头），可以这样组织：

1. 先输出全部 **8 条 extreme + 11 条 synergies + 5 条 allround = 24 个文件**
2. 再分 4 批输出 intervals：`000-063`、`064-127`、`128-191`、`192-255`，每批 64 条

每批末尾要附一行 `---DONE: <类别>---` 表示该类完成。

## 验证清单（生成后请自查）

- [ ] intervals 共 256 个文件，文件名 `000.json`–`255.json`
- [ ] 每个 intervals 的 `index` 与文件名数字一致、`key` 与 index 二进制位对应
- [ ] **256 个 intervals 的 label 不能有超过 5 个完全相同**（即 label 多样性 ≥ 251 种）
- [ ] extreme 共 8 个文件，`dim` / `letter` 严格按 8 维表
- [ ] **绝对没出现 `umami`、`鲜`** 作为字段名或第 5 维中文名
- [ ] synergies 字母对都是字典序排列（不是 `x-l` 而是 `l-x`）
- [ ] **没有破折号**（— / –），中文全角标点
- [ ] 没有 AI 味套话（"在……中"、"成就了……"、"是……的代表"）
- [ ] 每条 copy 长度合规（极档 / 联动 / 全能：15–35 字；区间：15–50 字）
- [ ] 全部为合法 JSON，UTF-8
