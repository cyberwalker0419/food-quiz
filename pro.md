# 味觉测试项目：算法架构设计与全面重构方案

## 一、 算法核心策略设计（基于 8 维感官模型）

针对 8 个连续的感官维度（酸 `sour` / 甜 `sweet` / 苦 `bitter` / 辣 `spicy` / 咸 `salty` / 鲜 `umami` / 脆 `crunchy` / 嫩 `tender`），算法的核心思想是：**"底层构建 8 维高精度向量，表层通过区间映射生成区间文案与联动文案"**。

### 1. 结果评估层 (Result Evaluation Layer)
**核心作用**：接收答题数据，计算 8 维精确得分，转化为用户易懂的"八维图（雷达图）"、"区间文案"及"精准菜品推荐"。
*   **多标签加权与 Min-Max 归一化**：将原始总分映射到 `[0, 100]` 区间，生成标准 8 维味觉向量。
*   **区间文案映射**：每条文案绑定到一个"维度 + 档位"组合（详见 Phase 3.3-3.5）。
*   **混合相似度与反向排斥**：使用**标准余弦相似度**叠加**欧氏距离倒数**精准推荐菜品；利用最低分维度生成"避雷指南"。

### 2. 动态出题层 (Dynamic Questioning Layer)
**核心作用**：实时决定"下一题出什么"，高效探测 8 个维度，将题量控制在 10-12 题，避免疲劳。
*   **多标签加权矩阵**：每个选项携带 8 维权重向量，一道题同时更新多个维度。
*   **基于信息增益的动态路由 (CAT 自适应)**：实时计算 8 个维度的方差，优先展示能探测当前最不确定维度的题目。
*   **硬性剪枝 (Pruning)**：若用户初始表现出对某维度极度排斥，后续直接过滤相关题目。

### 3. 题库调度层 (Question Bank Scheduling Layer)
**核心作用**：后台管理题库，保证生成的题单科学覆盖 8 维。
*   **约束满足问题 (CSP)**：设定硬约束，数学上保证不漏测。
*   **随机洗牌与平行题单**：打散顺序，生成多套平行题单。

## 二、 现有方案 (algorithm.md) 痛点与对比分析

| 模块 | 现有方案痛点 | 重构优化方向 |
| :--- | :--- | :--- |
| **评估层** | Pearson 在向量平缓时失效，排名靠噪声。 | 改用 **标准余弦 + 欧氏距离惩罚**。 |
| **出题层** | 前 25% 强制纯随机，未做到及时止损。 | 引入 **信息增益决策树**，实现极致效率与剪枝。 |
| **调度层** | 修改历史答案后，后续题目无法重新自适应。 | 重构状态管理，回退修改时**清除后续题目的重排标记**。 |
| **结果层** | 标签单薄、缺乏传播力；维度采用英文命名不符合中文用户直觉；"基因码 + 原型彩蛋"两层结构繁复。 | 引入**中文 8 维（酸甜苦辣咸鲜脆嫩）+ 区间文案 + 联动文案**；摒弃"ABCD-pq 基因码"和"16 原型彩蛋"双层结构。 |

## 三、 极致重构执行指南 (Claude Code Prompt)

```text
Task: 味觉测试项目底层算法全面重构 (Project Refactoring)

Context & Goal:
之前的 algorithm.md 方案存在理论缺陷（Pearson 导致平庸菜系失效、出题效率低、结果缺乏社交传播力；且维度命名采用英文，中文用户感知生疏；标签机制采用 "ABCD-pq 基因码 + 16 原型彩蛋" 两层结构，维护成本高且覆盖率有限）。现采用**全新架构**：

  · 维度层：8 个中文味觉维度（酸甜苦辣咸鲜脆嫩）
  · 标签层：区间文案（256 条全组合）+ 极档特殊文案（8 条）+ 联动文案（约 10 条高频组合）
  · 评估层：Min-Max 归一化 + 标准余弦相似度 + 欧氏距离惩罚（沿用升级）
  · 推荐层：动态计算 Top N 菜品（独立模块，折叠在文案下方）

请按以下架构重构，出题层与混合相似度计算逻辑保留升级，结果层完全重做。

Core Architecture Shift (核心架构转变):
1. 维度层：从 8 个英文感官维度 → 8 个中文味觉维度（sour/sweet/bitter/spicy/salty/umami/crunchy/tender）。
2. 标签层：从"基因码 ABCD-pq + 16 原型彩蛋文案" → "区间文案 + 极档文案 + 联动文案"三元结构。
3. 评估层：Min-Max 归一化 + 标准余弦相似度 + 欧氏距离惩罚（沿用）。
4. 推荐层：新增"菜品推荐"模块，运行时计算，UI 折叠在文案下方。

请严格按照以下 3 个 Phase 分步执行，每完成一个 Phase 停下来 review。

---

Phase 1: 题库数据结构与多标签矩阵重构 (Data Structure)
目标：打破"一题测一维"的低效模式，实现"一题多测"，维度名改为中文。
要求：
1. 重新定义 Question 和 Option 的 TypeScript 接口，维度字段使用英文驼峰：`sour / sweet / bitter / spicy / salty / umami / crunchy / tender`。
2. 每个 Option 必须携带完整的 8 维权重向量（以 `Record<TasteDimension, number>` 形式）。
3. 提供题库 JSON 示例，展示如何配置多标签矩阵（覆盖 8 个维度的中高低三档）。

Phase 2: 动态出题引擎重构 (Adaptive Engine)
目标：用最少的题（10-12题）精准收敛 8 个维度的得分。
要求：
1. 废弃现有的 pickNextQuestion 逻辑。
2. 引入信息增益/不确定性降低策略：实时计算当前用户 8 维 profile 的方差/信息熵，找出最不确定的 1-2 个维度，挑选能最大程度探测它们的题目。
3. 引入硬性剪枝：若用户在前 3 题中明确表现出对某维度极度排斥，后续直接过滤该维度权重 > 0 的题目。
4. 写出新的 AdaptiveSelector 核心类或函数，维度名采用 Phase 1 的中文变量。

Phase 3: 评估与结果生成重构 (Evaluation & Result Generation)
目标：用区间文案和联动文案取代旧基因码+彩蛋方案，生成高网感的"中文味觉图谱"结果页。

要求：

1. **向量归一化**：将用户答题累加的原始 8 维向量，通过 Min-Max 归一化映射到 [0, 100] 区间。归一化后的向量记为 `V = (sour, sweet, bitter, spicy, salty, umami, crunchy, tender)`，每维 ∈ [0, 100]。

2. **区间阈值**（三档，可微调，后续若效果不佳可调整边界）：
   - **低档**：`[0, 60]`
   - **高档**：`(60, 90)`，包含极档
   - **极档**：`[90, 100]` ⊂ 高档，触发**额外**的特殊文案（追加在高档文案之上）

3. **八维图渲染**（雷达图）：雷达图 8 个轴的数值即为归一化后的 8 维向量，每轴长度按 [0, 100] 等比缩放。每轴外侧标注**该维度的当前档位名称**（如"高辣"、"低苦"），档位名称本身也是文案的一部分。

4. **区间文案**（基础，2^8 = 256 条）：
   - 每维根据"低档/高档"分两档（高档包含极档，但极档另触发极档文案），8 维共 **2^8 = 256 种高低组合**。
   - 每种组合对应**一条**预制文案，文案以 8 维布尔向量 `(H/L)^8` 作为索引键。
   - 文案示例：
     - `(L,L,L,L,L,L,L,L)` → "味觉素人,什么都行,什么都不挑"
     - `(H,L,L,L,L,L,L,L)` → "嗜酸星人,一口下去眉头都不皱"
     - `(H,L,H,L,L,L,H,L)` → "酸苦交织的硬核派"
     - `(H,H,H,H,H,H,H,H)` → "味觉深渊行者,什么都要最重"
   - 文案存放在 `food-quiz/src/content/intervals/` 目录下。允许选型：256 个单文件 JSON 或 1 个 mega JSON；选型由 Claude Code 在 PR 中说明。

5. **极档特殊文案**（n = 8 条，每维 1 条）：
   - 仅当某维度数值 `≥ 90` 时，触发该维度的"极档特殊文案"，作为该维度区间文案的**追加层**（不替换）。
   - 8 维各 1 条，共 **n = 8 条**，独立存放在 `food-quiz/src/content/extreme/` 目录。
   - 文案示例：`spicy ≥ 90` → "🔥 极辣警告:你的胃已提交辞呈"、"辣到灵魂出窍,筷子还在找"
   - 极档文案必须有足够强的传播力（短句、金句感、emoji 适度）。

6. **联动文案**（C(8,2) = 28 对高频组合，**默认生成约 10 条**,其余组合触发通用兜底文案）：
   - 仅当用户向量的 **Top 1 维度** 和 **Top 2 维度** 都满足"高档(> 60)"时触发，生成 **1 条**联动文案。
   - 联动文案独立成条，渲染在区间文案和极档文案之上（或最下方，由 UI 设计决定）。
   - 文案示例：
     - 辣 + 鲜 → "🔥 鲜辣双煞,味觉的'火上浇油'组合"
     - 酸 + 甜 → "酸甜交织,初恋的味道"
     - 苦 + 咸 → "苦咸老饕,黑咖啡加酱油的苦行者"
   - 8 维两两组合共 C(8,2) = 28 对，**不必全配**。优先配 Top 10 高频组合（传播力强的），其余留作"通用兜底"文案（一条通用联动模板）。
   - 存放在 `food-quiz/src/content/synergies/` 目录，按维度对命名（如 `spicy-umami.json`）。

7. **弱无序渲染**（高档优先 + 限量 + 完整版折叠）：
   - **默认视图**：只显示"高档维度对应的高档区间文案"中，按"该维度数值与 50 的绝对距离"**降序**排列的 **前 3 条**。若用户某维度达到极档（≥ 90），该维度的极档特殊文案优先占位 1 条。
   - **完整版**：点击"展开完整解读"后，按"绝对值距离 50 降序"展示**全部 8 维**区间文案 + 极档特殊文案（如有）+ 联动文案（如有）。
   - **弱无序**含义：不强制按维度名固定顺序（如不按"酸甜苦辣"字母序），而是按数值强度动态排序，体现"以用户最突出的特征为锚"的传播理念。

8. **前置判定**（味觉方差）：
   - 计算用户 8 维向量的标准差。若 < 15，判定为"味觉端水大师/混沌全能老饕"，此分支**不进入区间文案机制**，直接触发"全能文案"（单独一份，与 256 条区间文案不重叠）。
   - 全能文案可放 3-5 条（随机/轮播展示），强调"兼容性"。

9. **避雷指南**：提取最低分维度（如 `tender < 20`），生成反向建议，如"嫩度绝缘体,慎选溏心蛋、慕斯等软糯类"。

10. **菜品推荐**（独立模块，运行时计算）：
    - 菜品数据采用**单文件**形式存放：`food-quiz/src/content/dishes.json`，包含菜系向量表。
    - 数据结构示例：
      ```json
      [
        {
          "name": "麻辣火锅",
          "cuisine": "川菜",
          "vector": { "sour": 20, "sweet": 15, "bitter": 10, "spicy": 95, "salty": 70, "umami": 85, "crunchy": 40, "tender": 50 }
        },
        {
          "name": "白切鸡",
          "cuisine": "粤菜",
          "vector": { "sour": 5, "sweet": 10, "bitter": 5, "spicy": 5, "salty": 40, "umami": 75, "crunchy": 15, "tender": 90 }
        }
      ]
      ```
    - 推荐公式（沿用 Phase 3.2）：`Score = α * Cosine(V, C) + β * (1 / (1 + Euclidean(V, C)))`，默认 `α = β = 0.5`。
    - 渲染位置：文案下方，**默认折叠**。点击"看看今天吃啥 ▼"后展开，显示按 Score 降序的 Top 5-8 道菜，每条带菜系标签。
    - 文案文件中**不附加 `dishHint` 字段**，避免与算法推荐结果重复/冲突。文案和菜品严格解耦。

11. **渲染顺序总览**（完整版）：
    ```
    [联动文案]               ← Top1+Top2 都高档时
    [八维图(雷达图)]          ← 永远显示
    [区间文案 ×8,按强度排序]   ← 默认仅前 3
    [极档特殊文案 ×k, k≤8]    ← 仅触发时显示
    [避雷指南]                ← 永远显示
    [▼ 推荐菜(可折叠)]        ← 永远显示（算法 Top N）
    ```
