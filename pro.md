## 一、 算法核心策略设计（基于 8 维感官模型）

针对 8 个连续的感官维度（酸 `sour` / 甜 `sweet` / 苦 `bitter` / 辣 `spicy` / 咸 `salty` / 鲜 `umami` / 脆 `crunchy` / 嫩 `tender`），算法的核心思想是：**"底层构建 8 维高精度向量，表层通过区间映射生成区间文案与联动文案"**。

### 1. 结果评估层 (Result Evaluation Layer)
**核心作用**：接收答题数据，计算 8 维精确得分，转化为用户易懂的"八维图（雷达图）"、"区间文案"及"精准菜品推荐"。
*   **多标签加权与 Min-Max 归一化**：将原始总分映射到 `[0, 100]` 区间，生成标准 8 维味觉向量。
*   **区间文案映射**：每条文案绑定到一个"维度 + 档位"组合（详见 Phase 3.3-3.5）。
*   **混合相似度与反向排斥**：使用**标准余弦相似度**叠加**欧氏距离倒数**精准推荐菜品；利用最低分维度生成"避雷指南"。

### 2. 动态出题层 (Dynamic Questioning Layer)
**核心作用**：实时决定"下一题出什么"，高效探测 8 个维度，将题量控制在 10-12 题，避免疲劳。
*   **多标签加权矩阵**：每个选项携带 8 维权重向量，一道题同时更新多个维度。
*   **基于信息增益的动态路由 (CAT 自适应)**：实时计算 8 个维度的方差，优先展示能探测当前最不确定维度的题目。
*   **硬性剪枝 (Pruning)**：若用户初始表现出对某维度极度排斥，后续直接过滤相关题目。

### 3. 题库调度层 (Question Bank Scheduling Layer)
**核心作用**：后台管理题库，保证生成的题单科学覆盖 8 维。
*   **约束满足问题 (CSP)**：设定硬约束，数学上保证不漏测。
*   **随机洗牌与平行题单**：打散顺序，生成多套平行题单。

## 二、 现有方案 (algorithm.md) 痛点与对比分析

| 模块 | 现有方案痛点 | 重构优化方向 |
| :--- | :--- | :--- |
| **评估层** | Pearson 在向量平缓时失效，排名靠噪声。 | 改用 **标准余弦 + 欧氏距离惩罚**。 |
| **出题层** | 前 25% 强制纯随机，未做到及时止损。 | 引入 **信息增益决策树**，实现极致效率与剪枝。 |

---

Phase 1: 题库数据结构与多标签矩阵重构 (Data Structure)

要求：

3. 引入硬性剪枝：若用户在前 3 题中明确表现出对某维度极度排斥，后续直接过滤该维度权重 > 0 的题目。
4. 写出新的 AdaptiveSelector 核心类或函数，维度名采用 Phase 1 的中文变量。

Phase 3: 评估与结果生成重构 (Evaluation & Result Generation)
目标：用区间文案和联动文案取代旧基因码+彩蛋方案，生成高网感的"中文味觉图谱"结果页。

要求：

1. **向量归一化**：将用户答题累加的原始 8 维向量，通过 Min-Max 归一化映射到 [0, 100] 区间。归一化后的向量记为 `V = (sour, sweet, bitter, spicy, salty, umami, crunchy, tender)`，每维 ∈ [0, 100]。

2. **区间阈值**（三档，可微调，后续若效果不佳可调整边界）：
   - **低档**：`[0, 60]`
   - **高档**：`(60, 90)`，包含极档
   - **极档**：`[90, 100]` ⊂ 高档，触发**额外**的特殊文案（追加在高档文案之上）

3. **八维图渲染**（雷达图）：雷达图 8 个轴的数值即为归一化后的 8 维向量，每轴长度按 [0, 100] 等比缩放。每轴外侧标注**该维度的当前档位名称**（如"高辣"、"低苦"），档位名称本身也是文案的一部分。

4. **区间文案**（基础，2^8 = 256 条）：
   - 每维根据"低档/高档"分两档（高档包含极档，但极档另触发极档文案），8 维共 **2^8 = 256 种高低组合**。
   - 每种组合对应**一条**预制文案，文案以 **8 字符单字母索引串** 作为索引键。
   - **字母映射表**（顺序固定，不允许重排）：

     | 位置 | 维度 | 字段 | 拼音 | 单字母 | 备注 |
     |:---:|:---:|:---:|:---:|:---:|:---|
     | 1 | 酸 | sour | suan | **S** | 首字母 |
     | 2 | 甜 | sweet | tian | **T** | 首字母 |
     | 3 | 苦 | bitter | ku | **K** | 首字母 |
     | 4 | 辣 | spicy | la | **L** | 首字母 |
     | 5 | 咸 | salty | xian | **I** | 第 2 字母，因「鲜」占 X 让位 |
     | 6 | 鲜 | umami | xian | **X** | 优先保留首字母 |
     | 7 | 脆 | crunchy | cui | **C** | 首字母 |
     | 8 | 嫩 | tender | nen | **N** | 首字母 |

     > 唯一冲突是「咸/鲜」（拼音同 xian），约定：**鲜保留 X，咸用第 2 字母 I**。8 字母 `S T K L I X C N` 全部互异，保证 256 个组合字符串唯一。
   - **大小写规则**：**大写 = 高档(> 60，含极档)，小写 = 低档(≤ 60)**。极档（≥ 90）不改变字母大小写，仅在区间文案之上**额外追加**极档文案，所以一个索引串仍唯一对应一条区间文案。
   - **索引示例**：
     - `stklixcn`（全低）→ "味觉素人,什么都行,什么都不挑"
     - `Stklixcn`（仅酸高）→ "嗜酸星人,一口下去眉头都不皱"
     - `StKliXcN`（酸、苦、脆高）→ "酸苦交织的硬核派"
     - `STKLIXCN`（全高）→ "味觉深渊行者,什么都要最重"
   - **Windows 兼容（NTFS 大小写不敏感）**：磁盘文件名不得直接使用混合大小写索引串（`StKliXcN.json` 与 `stklixcn.json` 会被视为同一文件）。**采用"逻辑 key + 序号"双轨制**：
     - 逻辑 key（代码 / JSON 字段内）：8 字符单字母串，如 `StKliXcN`
     - 存储文件名：`000.json`–`255.json`（3 位十进制序号）
     - 互转规则：S T K L I X C N 顺序固定为 bit7..bit0，大写→1、小写→0，按位拼成 8-bit 二进制再转十进制
   - **存储目录与文件结构**：

     ```
     food-quiz/src/content/intervals/
       000.json   ← key="stklixcn" → 味觉素人
       001.json   ← key="stklixcN" → 仅脆高
       ...
       163.json   ← key="StKliXcN" → 酸苦交织的硬核派
       ...
       255.json   ← key="STKLIXCN" → 味觉深渊行者
     ```

     每个 JSON 内部示例：

     ```json
     {
       "index": 163,
       "key": "StKliXcN",
       "label": "酸苦交织的硬核派",
       "copy": "酸与苦的极限拉扯,是你对平庸菜系的公然挑衅"
     }
     ```

   - **核心互转函数**（`food-quiz/src/lib/taste/keys.ts`，代码与文案共享此唯一映射）：

     ```ts
     export const DIMS = ['S','T','K','L','I','X','C','N'] as const;

     export function keyToIndex(key: string): number {
       return parseInt(
         key.split('').map(c => c === c.toUpperCase() ? '1' : '0').join(''),
         2
       );
     }

     export function indexToKey(index: number): string {
       return DIMS.map((c, i) => (index >> (7 - i)) & 1 ? c : c.toLowerCase()).join('');
     }
     ```

   - 选型决定：**256 个独立 JSON 文件**（序号命名），便于单人 PR 中按位补齐 / 修改文案而不引发 JSON 合并冲突。

5. **极档特殊文案**（n = 8 条，每维 1 条）：
   - 仅当某维度数值 `≥ 90` 时，触发该维度的"极档特殊文案"，作为该维度区间文案的**追加层**（不替换）。
   - 8 维各 1 条，共 **n = 8 条**，独立存放在 `food-quiz/src/content/extreme/` 目录。
   - **解耦设计**：极档文案文件**不依赖区间文案、联动文案或菜品数据**。每条文件只描述"该维度 ≥ 90 时说什么"，不引用任何 index / key / 菜品向量。
   - **文件名（使用 P3.4 的 8 字母单字母体系）**：`<单字母小写>.json`，8 个文件一一对应：

     | 单字母 | 维度 | 文件名 |
     |:---:|:---:|:---:|
     | s | 酸 sour | `s.json` |
     | t | 甜 sweet | `t.json` |
     | k | 苦 bitter | `k.json` |
     | l | 辣 spicy | `l.json` |
     | i | 咸 salty | `i.json` |
     | x | 鲜 umami | `x.json` |
     | c | 脆 crunchy | `c.json` |
     | n | 嫩 tender | `n.json` |

   - **结构示例**（`food-quiz/src/content/extreme/l.json`）：

     ```json
     {
       "dim": "spicy",
       "letter": "l",
       "label": "极辣",
       "threshold": 90,
       "copy": [
         "🔥 极辣警告:你的胃已提交辞呈",
         "辣到灵魂出窍,筷子还在找"
       ]
     }
     ```

   - **设计要点**：
     - `copy` 为**数组**（同一极档可轮播多条备选文案，由运行时随机选取），但**条目数固定 1–3**。
     - 文件名一律**小写**，避免 Windows NTFS 大小写敏感性问题。
     - 极档文件**不引用**区间文案的 `key` / `index`，亦不引用联动 / 菜品数据；删除任意极档文件不影响其他模块。
   - 极档文案必须有足够强的传播力（短句、金句感、emoji 适度）。

6. **联动文案**（C(8,2) = 28 对高频组合，**默认生成约 10 条**,其余组合触发通用兜底文案）：
   - 仅当用户向量的 **Top 1 维度** 和 **Top 2 维度** 都满足"高档(> 60)"时触发，生成 **1 条**联动文案。
   - 联动文案独立成条，渲染在区间文案和极档文案之上（或最下方，由 UI 设计决定）。
   - **解耦设计**：联动文案文件**不依赖区间文案的 256 条组合，不引用菜品数据**。每条联动文件只描述"某对维度同时高档时说什么"，独立加载、独立渲染。
   - **文件名（按字母对排序，单字母 + 连字符）**：`<字母A小写>-<字母B小写>.json`，字母固定按 `S T K L I X C N` 字典序排列，确保同名只对应唯一一对：

     | 字母对 | 维度组合 | 文件名 |
     |:---:|:---|:---:|
     | S–T | 酸 + 甜 | `s-t.json` |
     | L–X | 辣 + 鲜 | `l-x.json` |
     | K–I | 苦 + 咸 | `k-i.json` |
     | … | … | `s-l.json`、`t-x.json`… |

     > 约定：所有文件名一律**小写**（如 `l-x.json` 而非 `L-X.json`），避免 Windows NTFS 大小写敏感性问题。
     > C(8,2) = 28 对，配齐后目录共 28 + 1（通用兜底）= 29 个文件。

   - **结构示例**（`food-quiz/src/content/synergies/l-x.json`）：

     ```json
     {
       "pair": ["spicy", "umami"],
       "letters": ["l", "x"],
       "label": "鲜辣双煞",
       "copy": [
         "🔥 鲜辣双煞,味觉的'火上浇油'组合",
         "辣椒遇见鲜,舌头在开派对"
       ]
     }
     ```

   - **通用兜底文案**（`food-quiz/src/content/synergies/_fallback.json`）：

     - 顶级联动文件中**未配对**的 18 种组合（共 28 对，约定只配 Top 10 高频）触发通用兜底。
     - **独立文件**，不与具体维度对绑定，**不引用**任何具体维度名。
     - 文件名前缀 `_` 表示"系统级"且排序最前，加在 `synergies/` 目录内：

       ```json
       {
         "id": "_fallback",
         "label": "强强联合",
         "template": "你最强的两个维度正在组队:{a} 与 {b},味觉界的新组合诞生了",
         "copy": [
           "你最强的两个维度正在组队:{a} 与 {b}",
           "{a}+{b} 的化学反应,让味蕾重新洗牌",
           "当 {a} 遇上 {b},连厨房的锅都在鼓掌"
         ]
       }
       ```

     - 模板占位符 `{a}` `{b}` 运行时由 **P3.4 维度中文名**（"酸/甜/苦/辣/咸/鲜/脆/嫩"）替换，**不引用区间文案的 index/key**。
   - 8 维两两组合共 C(8,2) = 28 对，**不必全配**。优先配 Top 10 高频组合（传播力强的），其余 18 对由 `_fallback.json` 兜底。
   - 运行时加载逻辑：先按 `letters` 匹配配对文件；未命中则读 `_fallback.json` 模板并填入维度名。

7. **弱无序渲染**（高档优先 + 限量 + 完整版折叠）：
   - **默认视图**：在 256 条 P3.4 区间文案中，只取**对应维度实际为高档（> 60,大写位）的条目**，按"该维度数值与 50 的绝对距离"**降序**排列的 **前 3 条**。若用户某维度达到极档（≥ 90），该维度的 P3.5 极档特殊文案优先占位 1 条。

     - 索引位为小写 → `低<中文名>`，如 `s` → "低酸"、"k" → "低苦"

   - **弱无序**含义：不强制按维度名固定顺序（如不按"酸甜苦辣"字母序），而是按数值强度动态排序，体现"以用户最突出的特征为锚"的传播理念。
